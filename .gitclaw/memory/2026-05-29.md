# 2026-05-29

- GitClaw v0 issue/comment assistant works in the main repository.
- `openai/gpt-5-mini` works through GitHub Models in Actions.
- Issue backups are committed to the `gitclaw-backups` branch after successful turns.
- Heartbeat should be implemented as a scheduled/manual GitHub Actions workflow,
  not a server. It scans `gitclaw:heartbeat` issues and uses same-slot
  idempotency.
- Read-only tools should include bounded search as `gitclaw.search_files`, with
  matching lines inserted as auditable prompt context before model inference.
- Issue-chat turns should manage lightweight status labels:
  `gitclaw:running`, `gitclaw:done`, and `gitclaw:error`.
- Failed started turns should post a bounded `gitclaw:error` diagnostic comment
  that points to the Actions run without leaking issue tokens or raw provider
  response bodies.
- Prompt assembly should enforce byte budgets before model calls: preserve the
  original issue and recent transcript tail, truncate large bodies visibly, and
  cap the final prompt.
- Prompt artifacts should be opt-in, redacted before upload, and stored as
  GitHub Actions artifacts rather than issue comments or logs.
- Write intent should be detected deterministically in read-only mode, labeled
  `gitclaw:write-requested`, and answered as a proposal rather than as a
  completed file change.
- Main issue handling can be woken with `workflow_dispatch` using
  `issue_number` plus a stable `dispatch_id`; this is the bridge-friendly path
  for scheduled Telegram/Slack pollers because `GITHUB_TOKEN`-authored comments
  should not be expected to recursively trigger `issue_comment` workflows.
- Mirrored channel comments should start with a hidden
  `gitclaw:channel-message` marker. GitClaw reconstructs those comments as
  untrusted user transcript messages even if a bridge posted them as
  `github-actions[bot]`.
- Generic channel ingest should be a workflow-dispatchable path: normalized
  channel/thread/message fields create or reuse a canonical
  `gitclaw:channel-thread` issue, post a `gitclaw:channel-message` comment,
  then dispatch the main handler with `<channel>-<message_id>`.
- Proactive usefulness belongs in the spec as scheduled GitHub Actions
  workflows that create or reuse their own `gitclaw:proactive-run` issues and
  then dispatch the normal handler. Do not silently let the agent install new
  cron workflows without review.
- Proactive enqueue is now the first implemented primitive: it creates/reuses
  one issue per proactive `name + slot`, writes a `gitclaw:proactive-run`
  marker, applies `gitclaw:proactive`, and dispatches the normal handler with
  `proactive-<name>-<slot>`.
- GitHub Models can return bursty `429` responses during dense live E2E runs;
  the OpenAI-compatible client now retries transient `429`, timeout, and `5xx`
  failures and uses a bounded HTTP timeout before posting the safe
  provider-failure comment. `Retry-After` delays are capped so rate-limit
  cooldowns cannot park an Actions worker indefinitely.
- `@gitclaw /context` is the deterministic context-inspection command. It
  assembles normal repo context, posts a `model="gitclaw/context"` report with
  context file, skill, and tool-output names/sizes, and avoids a model call.
- `@gitclaw /skills` is the deterministic local skill inventory command. It
  posts a `model="gitclaw/skills"` report with available and selected skill
  metadata, without dumping full `SKILL.md` bodies or calling a model.
- The `gitclaw-backups` branch now keeps a repo-scoped backup index beside raw
  issue JSON backups: `.gitclaw/backups/<owner>__<repo>/index.json` and
  `README.md`, with issue metadata and counts but no raw message bodies.
