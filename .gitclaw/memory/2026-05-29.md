# 2026-05-29

## Core Shape

- GitClaw is a repo-native GitHub issue/comment assistant running entirely from
  GitHub Actions. A GitHub issue is the conversation; issue comments continue
  the thread.
- `openai/gpt-5-mini` works through GitHub Models in Actions.
- Issue runs manage lightweight labels: `gitclaw:running`, `gitclaw:done`,
  `gitclaw:error`, `gitclaw:disabled`, and `gitclaw:write-requested`.
- Failed started turns post a bounded `gitclaw:error` diagnostic with an
  Actions run pointer, without leaking issue tokens or raw provider responses.
- Prompt assembly preserves the original issue plus recent transcript tail,
  applies byte budgets, and can upload redacted prompt artifacts as Actions
  artifacts.
- Write intent is detected deterministically in read-only mode and answered as
  proposal/analysis rather than executed file mutation.

## Wakeups And Channels

- Heartbeat is a scheduled/manual GitHub Actions workflow, not a daemon. It
  scans `gitclaw:heartbeat` issues and uses same-slot idempotency.
- Main issue handling can be woken with `workflow_dispatch` using
  `issue_number` plus a stable `dispatch_id`; this is the bridge-friendly path
  for scheduled Telegram/Slack pollers.
- Mirrored channel comments use a hidden `gitclaw:channel-message` marker and
  are reconstructed as untrusted user transcript messages even if posted by
  `github-actions[bot]`.
- Generic channel ingest creates/reuses a `gitclaw:channel-thread` issue,
  posts a mirrored channel message, then dispatches the normal handler with a
  channel/message idempotency key.
- `gitclaw channels list` mirrors `/channels` locally without issue-only
  marker counts; `/channels verify` and `gitclaw channels verify` add a
  body-free health check for workflow_dispatch, write permissions, normalized
  inputs, provider keys, and dispatch-id shape.

## Deterministic Reports

- `@gitclaw /context` reports loaded context files, selected skills, and
  deterministic tool-output metadata without a model call. `gitclaw context
  list` is the local body-free context inventory.
- `@gitclaw /prompt` reports provider/model, prompt size/hash, budgets,
  transcript truncation counts, context files, selected skills, and tool-output
  metadata without a model call. `@gitclaw /prompt list` is an explicit issue
  alias, and `gitclaw prompt list` is the local body-free prompt inventory.
- `@gitclaw /memory` and `/memory list` report memory inventory, counts,
  loaded/omitted state, hashes, and validation rollups without bodies.
- `@gitclaw /memory validate` reports body-free memory hygiene; `/memory verify`
  adds repo-local provenance, hashes, read-only status, and external memory
  provider/session-index/background-promotion non-goals.
- `@gitclaw /memory search <query>` searches memory files locally and reports
  query hashes, scores, loaded state, and file/line hashes without snippets.
- `@gitclaw /soul` reports loaded soul, identity, user, tools, memory,
  heartbeat, and dated memory files with sizes, hashes, and soul validation
  rollups. `@gitclaw /soul list` and `gitclaw soul list` are explicit inventory
  aliases.
- `@gitclaw /tools` reports deterministic read-only/metadata tool contracts,
  active tool-output metadata, guidance files, and validation rollups.
  `@gitclaw /tools list` and `gitclaw tools list` are explicit inventory
  aliases.
- `@gitclaw /skills` reports available and selected local skill metadata,
  hashes, frontmatter/description presence, requirement counts, and validation
  rollups. `@gitclaw /skills list` and `gitclaw skills list` are explicit
  inventory aliases.
- `@gitclaw /skills info <name>` and `gitclaw skills info <name>` render one
  focused skill metadata report without dumping full `SKILL.md` bodies.
- `@gitclaw /skills search <query>` and `gitclaw skills search <query>` search
  local skill metadata only and report query hash/term count, match fields,
  selection state, hashes, sizes, and requirement counts.
- `@gitclaw /soul search <query>` and `gitclaw soul search <query>` search
  loaded high-authority context files and report only query hashes, paths,
  categories, line numbers, scores, and file/line hashes.
- `@gitclaw /tools search <query>` and `gitclaw tools search <query>` search
  declared tool contracts and active-output metadata without echoing raw tool
  inputs or outputs.
- `@gitclaw /session` reports reconstructed issue-session counts, markers,
  trust states, message metadata, and hashes. `@gitclaw /session search
  <query>` searches the reconstructed issue transcript and returns only
  message indexes, sources, trust metadata, scores, and hashes. `@gitclaw
  /session list` is an explicit issue alias, and `gitclaw session list
  --backup <issue.json>` inspects a backed-up session locally without dumping
  issue, comment, assistant, or transcript bodies. `gitclaw session search
  <query> --backup <issue.json>` searches that backed-up transcript with the
  same body-free hash/count output.
- `@gitclaw /backup` reports the backup branch, issue JSON path, repo
  index/README paths, schema version, and current transcript counts; the
  post-turn backup job writes the canonical raw backup.
- `@gitclaw /models`, `/config`, `/doctor`, `/policy`, `/prompt`,
  `/budget`, `/prompt-budget`, `/channels`, `/proactive`, `/cron`, `/help`,
  and `/commands` are deterministic, body-free control-plane reports.
- `gitclaw models list` locally audits GitHub Models/OpenAI-compatible wiring:
  provider, model ID, endpoint host, token-source name, timeout, and retries.
- `gitclaw config list` locally audits effective config, labels, trusted
  associations, prompt budgets, command surface, and workflow file hashes.
- `gitclaw policy list` locally audits trusted associations, managed labels,
  expected workflow permissions, model/run mode, and policy-output metadata.
- `@gitclaw /doctor list` is an explicit issue alias, and `gitclaw doctor
  list` locally audits the same body-free health rollup without issue-only
  fields.
- `@gitclaw /tools validate` is the issue-side mirror of
  `gitclaw tools validate`, returning only body-free tool contract/output
  validation counts and findings.
- `@gitclaw /tools verify` and `gitclaw tools verify` expose the body-free
  deterministic tool trust envelope with contract modes, guidance provenance,
  active-output hashes, raw input suppression, and verification findings.
- `@gitclaw /skills validate` is the issue-side mirror of
  `gitclaw skills validate`, returning only body-free skill validation counts
  and findings.
- `@gitclaw /skills verify` and `gitclaw skills verify` expose a body-free
  repo-local skill trust envelope with source roots, hashes, requirement
  status, validation rollup, and `registry_verification=not_configured`.
- `@gitclaw /skills check` and `gitclaw skills check` are OpenClaw-style
  compatibility aliases for the same skills validation report.
- `@gitclaw /soul validate` is the issue-side mirror of
  `gitclaw soul validate`, returning only body-free high-authority context
  validation counts and findings.
- `@gitclaw /soul verify` and `gitclaw soul verify` expose the body-free
  repo-local trust envelope for high-authority context: source counts,
  required-file coverage, soul frontmatter/description status, trust cards,
  hashes, and explicit registry/profile verification non-goals.
- The `/help` E2E asserts every advertised local helper, including backup,
  proactive, memory, skills, soul, and tools helpers.

## Backups

- Issue backups are committed to the `gitclaw-backups` backup branch after
  successful turns.
- The backup branch keeps a repo-scoped index beside raw issue JSON backups:
  `.gitclaw/backups/<owner>__<repo>/index.json` and `README.md`, with issue
  metadata and counts but no raw message bodies.
- `gitclaw backup verify --root .gitclaw/backups --repo <owner/repo>`
  validates the fetched backup branch: repo index, README, traversal-safe
  canonical issue JSON paths, schema/count/timestamp consistency, and absence
  of unindexed issue backup files.
- `gitclaw backup manifest --root .gitclaw/backups --repo <owner/repo>
  --issue <n>` renders a file-level backup manifest with control-file and
  issue-payload hashes, counts, and byte sizes, without dumping raw bodies.
- `gitclaw backup stats --root .gitclaw/backups --repo <owner/repo>` verifies
  and summarizes a fetched backup branch with aggregate issue/comment/message
  counts, event type counts, latest backup metadata, and payload bytes.
- `gitclaw backup list --root .gitclaw/backups --repo <owner/repo> --limit <n>`
  verifies a fetched backup branch and lists newest indexed backups with issue
  numbers, paths, timestamps, counts, event names, and title hashes only.
- `gitclaw backup export-jsonl --root .gitclaw/backups --repo <owner/repo>
  --issue <n>` exports backed-up reconstructed transcript messages as raw JSONL
  recovery records. It is intentionally local CLI output, not issue-visible.
- `gitclaw backup restore-plan --root .gitclaw/backups --repo <owner/repo>
  --issue <n>` renders a non-mutating restore dry-run with metadata, counts,
  and body hashes.
- `gitclaw backup retention-plan --root .gitclaw/backups --repo <owner/repo>
  --keep-latest <n>` renders a non-mutating retention dry-run with kept/prune
  candidates and title hashes.
- `gitclaw backup search --root .gitclaw/backups --repo <owner/repo> <query>`
  verifies and searches a fetched `gitclaw-backups` tree locally, reporting
  only issue paths, source metadata, trust, scores, and body/line hashes.
- Issue-side `/backup <subcommand>` is metadata-only intent. It records the
  requested local backup command and backup paths, then the post-turn backup
  job writes `gitclaw-backups`; raw bodies and search queries stay out of
  comments.

## Proactive Work

- Proactive usefulness is scheduled GitHub Actions that create/reuse their own
  `gitclaw:proactive-run` issues and dispatch the normal handler.
- `gitclaw proactive enqueue` creates/reuses one issue per proactive
  `name + slot`, writes the marker, applies `gitclaw:proactive`, and dispatches
  the normal handler.
- `gitclaw proactive init --name <name> --cron <cron>` generates a proactive
  prompt file and scheduled workflow as ordinary repo files, supports
  `--dry-run`, refuses differing overwrites without `--force`, and reports
  hashes/status without dumping prompt bodies.
- `@gitclaw /proactive list` is an explicit issue alias, and `gitclaw
  proactive list` locally reports proactive workflow and prompt-file metadata
  without dumping prompt bodies or issue-only fields.
- The generic proactive workflow has a weekly scheduled path plus a safe
  `.gitclaw/proactive/repo-hygiene.md` default prompt.

## Runtime Notes

- Read-only tools include bounded `gitclaw.list_files`, `gitclaw.search_files`,
  `gitclaw.read_file`, `gitclaw.skill_index`, and `gitclaw.policy` metadata.
- Channel ingest dedupes repeated `channel + message_id` retries, dispatch
  wakeups read the matching `gitclaw:channel-message` as active text, and
  channel workflows store offsets, leases, and outbound delivery receipts as hashes.
- Skills are loaded with progressive disclosure; full `SKILL.md` bodies are
  prompt inputs only when selected or marked always-on.
- `.gitclaw/config.yml` is read-only, schema-checked repo config. Defaults load
  first, repo config second, and environment overrides last.
- GitHub Models can return bursty `429` responses during dense live E2E runs;
  the client retries transient `429`, timeout, and `5xx` failures with bounded
  delay and request timeout.
- Workflows use Node 24-compatible first-party actions: `actions/checkout@v5`,
  `actions/setup-go@v6`, and `actions/upload-artifact@v6`.
