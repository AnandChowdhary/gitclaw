# 2026-05-29

## Core Shape

- GitClaw is a GitHub Actions issue/comment assistant: issues are
  conversations, comments are turns, and there is no always-on server in the
  core loop.
- GitHub Models is the hosted default. The current repo default is
  `openai/gpt-5-nano`; GPT-5-family requests use `max_completion_tokens`.
- Issue runs manage `gitclaw:running`, `gitclaw:done`, `gitclaw:error`,
  `gitclaw:disabled`, and `gitclaw:write-requested`.
- Failed started turns post bounded `gitclaw:error` diagnostics with run links,
  without leaking issue tokens or raw provider responses.
- Prompt assembly preserves the issue plus recent transcript tail, applies byte
  budgets, and can upload redacted prompt artifacts.
- Write intent is detected deterministically in read-only mode and answered as
  proposal/analysis rather than file mutation.

## Wakeups And Channels

- Heartbeat is a scheduled/manual GitHub Actions workflow, not a daemon. It
  scans `gitclaw:heartbeat` issues and uses same-slot idempotency.
- Main issue handling can be woken with `workflow_dispatch` using
  `issue_number` plus a stable `dispatch_id`; this is the bridge path for
  scheduled Telegram/Slack pollers.
- Mirrored channel comments use hidden `gitclaw:channel-message` markers and
  are reconstructed as untrusted user transcript messages even if posted by
  `github-actions[bot]`.
- Generic channel ingest creates/reuses a `gitclaw:channel-thread`, mirrors the
  message, then dispatches the handler with a channel/message idempotency key.
- `/channels` list/verify/info/risk and local mirrors expose provider metadata,
  workflow dispatch shape, write permissions, inputs, dispatch IDs, and hashed
  offsets/receipts without dumping channel bodies.

## Deterministic Reports

- Deterministic slash commands run after preflight/context assembly but before
  model inference. They post `gitclaw:assistant-turn` comments with
  `model="gitclaw/<surface>"`, expose counts, paths, names, hashes, risk codes,
  validation totals, and status, and never dump issue/comment bodies, prompt
  text, context bodies, tool outputs, skill bodies, or secrets.
- The main report families are context, prompt, memory, soul, tools, skills,
  bundles, orders, hooks, plugins, tasks, agents, nodes, channels, backups,
  sessions, models, config, doctor, policy, approvals, artifacts, checkpoints,
  rollback, diffs, workspace, profile, sandbox, runs, migration, proactive, and
  heartbeat.
- Local `list`, `verify`, `validate`, `risk`, `info`, and `search` mirrors make
  most issue-side reports inspectable before opening an issue.
- Risk reports scan sensitive bodies internally when required, but issue-visible
  output stays body-free. Every implementation batch touching a risk or
  prompt-visible surface needs a real GitHub Models conversation E2E after the
  deterministic report.
- `/session` and local `gitclaw session ... --backup <issue.json>` are the
  provenance surface: they report assistant markers, trusted/untrusted message
  counts, prompt-context hashes, selected skill names, prompt-visible tool
  names, and transcript hashes without raw transcript bodies.
- `/backup <subcommand>` is metadata-only intent. Canonical raw issue backups
  are written after successful turns to the `gitclaw-backups` branch, while
  local backup commands verify, list, search, export, restore-plan,
  retention-plan, and coverage-check fetched backup trees.
- `/doctor` and `gitclaw doctor list` are the top-level OpenClaw/Hermes-style
  health rollups: config, workflow presence, context files, skills, memory,
  tools, proactive prompts, validation totals, and now E2E harness coverage.

## Proactive Work

- Proactive usefulness is scheduled GitHub Actions that create/reuse their own
  `gitclaw:proactive-run` issues and dispatch the normal handler.
- `gitclaw proactive enqueue` creates/reuses one issue per `name + slot`,
  writes the marker, applies `gitclaw:proactive`, and dispatches the handler.
- `gitclaw proactive init --name <name> --cron <cron>` generates a prompt file
  and scheduled workflow as reviewed repo files, supports `--skill`,
  `--dry-run`, overwrite refusal, and body-free hash/status reports.
- `/proactive list`, `/proactive info <name>`, `/proactive risk`, and local
  mirrors expose workflow, prompt, skill hints, schedule, and risk metadata
  without prompt bodies. Reminders use `--not-before` due gates.

## Runtime Notes

- Read-only built-in tools include bounded file list/search/read, skill index,
  and policy/config metadata. Search-result verification tokens must come from
  `gitclaw.search_files`, not issue-thread nonce echoes.
- Skills and tool outputs are repo-gated with allow/disable config. Full skill
  bodies are prompt inputs only when selected or always-on.
- `.gitclaw/config.yml` is read-only and schema-checked. Defaults load first,
  repo config second, environment overrides last.
- GitHub Models can return bursty `429` responses during dense live E2E; the
  client retries transient `429`, timeout, and `5xx` failures with bounded
  delay and request timeout.
- Normal LLM assistant markers carry body-free prompt provenance:
  prompt-context hash, context/skill/tool counts, selected skill names, and
  prompt-visible tool names so live E2E can prove inference and tool grounding.
- Workflows use Node 24-compatible first-party actions: `actions/checkout@v5`,
  `actions/setup-go@v6`, and `actions/upload-artifact@v6`.
- Artifact, diff, and workspace governance is body-safe: reports audit metadata,
  policy files, workflow versions, gates, counts, paths, hashes, and risk codes
  without printing prompt artifacts, patches, file bodies, or workflow bodies.
- Live E2E harnesses should create real GitHub issues in the main repo, close or
  retain them explicitly, and for feature batches include a normal GitHub
  Models turn proving prompt provenance and prompt-visible tool usage.
