# 2026-05-29

- GitClaw v0 issue/comment assistant works in the main repository.
- `openai/gpt-5-mini` works through GitHub Models in Actions.
- Issue backups are committed to the `gitclaw-backups` branch after successful turns.
- Heartbeat should be implemented as a scheduled/manual GitHub Actions workflow,
  not a server. It scans `gitclaw:heartbeat` issues and uses same-slot
  idempotency.
- Read-only tools should include bounded search as `gitclaw.search_files`, with
  matching lines inserted as auditable prompt context before model inference.
- Issue-chat turns should manage lightweight status labels:
  `gitclaw:running`, `gitclaw:done`, and `gitclaw:error`.
- Failed started turns should post a bounded `gitclaw:error` diagnostic comment
  that points to the Actions run without leaking issue tokens or raw provider
  response bodies.
- Prompt assembly should enforce byte budgets before model calls: preserve the
  original issue and recent transcript tail, truncate large bodies visibly, and
  cap the final prompt.
- Prompt artifacts should be opt-in, redacted before upload, and stored as
  GitHub Actions artifacts rather than issue comments or logs.
- Write intent should be detected deterministically in read-only mode, labeled
  `gitclaw:write-requested`, and answered as a proposal rather than as a
  completed file change.
- Main issue handling can be woken with `workflow_dispatch` using
  `issue_number` plus a stable `dispatch_id`; this is the bridge-friendly path
  for scheduled Telegram/Slack pollers because `GITHUB_TOKEN`-authored comments
  should not be expected to recursively trigger `issue_comment` workflows.
- Mirrored channel comments should start with a hidden
  `gitclaw:channel-message` marker. GitClaw reconstructs those comments as
  untrusted user transcript messages even if a bridge posted them as
  `github-actions[bot]`.
