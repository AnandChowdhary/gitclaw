# 2026-05-29

## Core Shape

- GitClaw is a GitHub Actions issue/comment assistant: issues are
  conversations, comments are turns.
- `openai/gpt-5-nano` is the default GitHub Models ID; the authenticated
  catalog currently exposes it while not exposing `openai/gpt-5.4-mini`.
- GPT-5-family GitHub Models requests use `max_completion_tokens`, not
  legacy `max_tokens`.
- Issue runs manage lightweight labels: `gitclaw:running`, `gitclaw:done`,
  `gitclaw:error`, `gitclaw:disabled`, and `gitclaw:write-requested`.
- Failed started turns post bounded `gitclaw:error` diagnostics with run links,
  without leaking issue tokens or raw provider responses.
- Prompt assembly preserves original issue + recent transcript tail, applies
  byte budgets, and can upload redacted prompt artifacts.
- Write intent is detected deterministically in read-only mode and answered as
  proposal/analysis rather than executed file mutation.

## Wakeups And Channels

- Heartbeat is a scheduled/manual GitHub Actions workflow, not a daemon. It
  scans `gitclaw:heartbeat` issues and uses same-slot idempotency.
- Main issue handling can be woken with `workflow_dispatch` using
  `issue_number` plus a stable `dispatch_id`; this is the bridge-friendly path
  for scheduled Telegram/Slack pollers.
- Mirrored channel comments use a hidden `gitclaw:channel-message` marker and
  are reconstructed as untrusted user transcript messages even if posted by
  `github-actions[bot]`.
- Generic channel ingest creates/reuses a `gitclaw:channel-thread`, mirrors the
  message, then dispatches the handler with a channel/message idempotency key.
- `/channels` list/verify/info and local mirrors expose bridge/provider
  metadata, workflow_dispatch, write perms, inputs, and dispatch IDs.

## Deterministic Reports

- `@gitclaw /context` reports loaded context files, selected skills, and
  deterministic tool-output metadata without a model call. `gitclaw context
  list` is the local body-free context inventory; `/context info <path>` and
  `gitclaw context info <path>` give one focused body-free context card.
- `@gitclaw /prompt` reports provider/model, prompt size/hash, budgets,
  transcript truncation counts, context files, selected skills, and tool-output
  metadata without a model call. `@gitclaw /prompt list` is an explicit issue
  alias, and `gitclaw prompt list` is the local body-free prompt inventory.
- `@gitclaw /memory` and `/memory list` report memory inventory, counts,
  loaded/omitted state, hashes, and validation rollups without bodies.
- `@gitclaw /memory validate` reports body-free memory hygiene; `/memory verify`
  adds repo-local provenance, hashes, read-only status, and external memory
  provider/session-index/background-promotion non-goals.
- `@gitclaw /memory risk` and `gitclaw memory risk` scan long-term and dated
  memory notes for prompt-boundary, credential, hidden-persistence, automation,
  and host-execution risks while reporting only counts, codes, paths, and line
  hashes. Risk-surface changes require a live GitHub Models E2E follow-up.
- `@gitclaw /memory search <query>` searches memory files locally and reports
  query hashes, scores, loaded state, and file/line hashes without snippets.
- `@gitclaw /soul` reports loaded soul, identity, user, tools, memory,
  heartbeat, and dated memory files with sizes, hashes, and soul validation
  rollups. `@gitclaw /soul list` and `gitclaw soul list` are explicit inventory
  aliases.
- `@gitclaw /tools` reports deterministic read-only/metadata tool contracts,
  active tool-output metadata, guidance files, and validation rollups.
  `@gitclaw /tools list` and `gitclaw tools list` are explicit inventory
  aliases.
- `@gitclaw /skills` reports available and selected local skill metadata,
  hashes, config gate state, frontmatter/description presence, requirement
  counts, and validation rollups. `skills.allowed` and `skills.disabled` keep
  blocked skill bodies out of prompts while preserving body-free metadata.
- `@gitclaw /skills info <name>` and `gitclaw skills info <name>` render one
  focused skill metadata report without dumping full `SKILL.md` bodies.
- `@gitclaw /skills search <query>` and `gitclaw skills search <query>` search
  local skill metadata only and report query hash/term count, match fields,
  selection state, hashes, sizes, and requirement counts.
- `@gitclaw /orders` and `gitclaw orders list|verify` audit
  `.gitclaw/STANDING_ORDERS.md`, standing-order clause coverage, model-context
  loading, and proactive enforcement metadata without executing orders or
  dumping order bodies.
- `@gitclaw /hooks` and `gitclaw hooks list|verify` audit
  `.gitclaw/HOOKS.md`, declarative `.gitclaw/hooks/*.md` specs,
  audit-only/approval gates, ignored executable handlers, and context loading
  without executing hooks or dumping hook bodies.
- `@gitclaw /hooks risk` and local `gitclaw hooks risk` scan hook policy,
  declarative hook specs, and ignored executable-looking handlers for
  prompt-boundary, credential, raw payload logging, untrusted issue-body
  execution, webhook, repository mutation, approval/audit-only, and
  unbounded-loop risks while reporting only metadata, codes, severities, and
  line hashes.
- `@gitclaw /plugins` and `gitclaw plugins list|verify` audit
  `.gitclaw/PLUGINS.md`, declarative `.gitclaw/plugins/*.md` specs,
  metadata-only activation, approval gates, ignored package/runtime files, and

- `@gitclaw /plugins risk` and local `gitclaw plugins risk` scan plugin

- `@gitclaw /tasks` and `gitclaw tasks list|verify` audit

  subagent, external task database, raw payload logging, untrusted issue-body

- `@gitclaw /session` reports reconstructed issue-session counts, markers,
  trust states, message metadata, and hashes. `@gitclaw /session search
  <query>` searches the reconstructed issue transcript and returns only

  issue, comment, assistant, or transcript bodies. `gitclaw session search
  <query> --backup <issue.json>` searches that backed-up transcript with the

- `@gitclaw /backup` reports the backup branch, issue JSON path, repo
  index/README paths, schema version, and current transcript counts; the
  post-turn backup job writes the canonical raw backup.
- `@gitclaw /models`, `/config`, `/doctor`, `/policy`, `/channels`,
  `/proactive`, `/help`, and aliases are deterministic body-free control-plane

- `@gitclaw /tools validate` is the issue-side mirror of
  `gitclaw tools validate`, returning only body-free tool contract/output
  validation counts and findings.
- `@gitclaw /tools verify` and `gitclaw tools verify` expose the body-free
  deterministic tool trust envelope with contract modes, guidance provenance,
  active-output hashes, raw input suppression, and verification findings.
- `@gitclaw /tools risk` and `gitclaw tools risk` expose the body-free
  OpenClaw/Hermes-inspired risk audit for contracts, guidance, active tool
  outputs, and active tool inputs, and require a live LLM-backed E2E after
  risk-surface changes.
- `@gitclaw /channels risk` and `gitclaw channels risk` expose a body-free

  severities, and hashes, and every change needs a live GitHub Models

- `@gitclaw /skills validate` is the issue-side mirror of
  `gitclaw skills validate`, returning only body-free skill validation counts

- `@gitclaw /skills verify` exposes the repo-local trust envelope with sources,
  hashes, gate state, requirements, validation, and no-registry status.

- `@gitclaw /soul validate` is the issue-side mirror of
  `gitclaw soul validate`, returning only body-free high-authority context

## Backups

- Issue backups are committed to the `gitclaw-backups` backup branch after
  successful turns; backup writes are serialized repo-wide so parallel issue
  runs do not race on the shared branch ref.
- The backup branch keeps a repo-scoped index beside raw issue JSON backups:
  `.gitclaw/backups/<owner>__<repo>/index.json` and `README.md`, with issue
  metadata and counts but no raw message bodies.
- `gitclaw backup verify --root .gitclaw/backups --repo <owner/repo>`
  validates the fetched backup branch: repo index, README, traversal-safe
  canonical issue JSON paths, schema/count/timestamp consistency, and absence
  of unindexed issue backup files.
- `gitclaw backup risk --root .gitclaw/backups --repo <owner/repo>` verifies
  and scans indexed backup payloads for integrity, path-safety,
  credential-handling, prompt-boundary, restore-safety, and retention risks
  while reporting only paths, counts, codes, severities, and line hashes.
- `@gitclaw /checkpoints risk`, `@gitclaw /rollback risk`, and local
  `gitclaw checkpoints risk` scan git checkpoint metadata for missing
  auditability, dirty worktrees, raw diff/file-body exposure,
  restore/reset/clean/checkout authority, shadow-store path leakage, and
  rollback safety gates while reporting only counts, commit hashes, codes, and
  severities.
- `gitclaw backup manifest --root .gitclaw/backups --repo <owner/repo>
  --issue <n>` renders a file-level backup manifest with control-file and
  issue-payload hashes, counts, and byte sizes, without dumping raw bodies.
- `gitclaw backup stats --root .gitclaw/backups --repo <owner/repo>` verifies
  and summarizes a fetched backup branch with aggregate issue/comment/message
  counts, event type counts, latest backup metadata, and payload bytes.
- `gitclaw backup list --root .gitclaw/backups --repo <owner/repo> --limit <n>`
  verifies a fetched backup branch and lists newest indexed backups with issue
  numbers, paths, timestamps, counts, event names, and title hashes only.
- `gitclaw backup info --root .gitclaw/backups --repo <owner/repo> --issue <n>`
  verifies one fetched backup payload and reports counts plus title/body hashes
  without dumping raw issue, comment, or transcript bodies.
- `gitclaw backup export-jsonl --root .gitclaw/backups --repo <owner/repo>
  --issue <n>` exports backed-up reconstructed transcript messages as raw JSONL
  recovery records. It is intentionally local CLI output, not issue-visible.
- `gitclaw backup restore-plan --root .gitclaw/backups --repo <owner/repo>
  --issue <n>` renders a non-mutating restore dry-run with metadata, counts,
  and body hashes.
- `gitclaw backup retention-plan --root .gitclaw/backups --repo <owner/repo>
  --keep-latest <n>` renders a non-mutating retention dry-run with kept/prune
  candidates and title hashes.
- `gitclaw backup search --root .gitclaw/backups --repo <owner/repo> <query>`
  verifies and searches a fetched `gitclaw-backups` tree locally, reporting
  only issue paths, source metadata, trust, scores, and body/line hashes.
- Issue-side `/backup <subcommand>` is metadata-only intent. It records the
  requested local backup command and backup paths, then the post-turn backup
  job writes `gitclaw-backups`; raw bodies and search queries stay out of
  comments.

## Proactive Work

- Proactive usefulness is scheduled GitHub Actions that create/reuse their own
  `gitclaw:proactive-run` issues and dispatch the normal handler.
- `gitclaw proactive enqueue` creates/reuses one issue per proactive
  `name + slot`, writes the marker, applies `gitclaw:proactive`, and dispatches
  the normal handler.
- `gitclaw proactive init --name <name> --cron <cron>` generates a proactive
  prompt file and scheduled workflow as ordinary repo files, supports
  `--skill`, `--dry-run`, overwrite refusal, and body-free hash/status reports.
- `/proactive list`/`info <name>` and local mirrors report workflow, prompt,
  skill hints, and job metadata without prompt bodies.
- `/proactive risk` and local `gitclaw proactive risk` scan proactive prompts
  and the generic scheduled workflow for prompt-boundary, credential, raw
  prompt logging, host execution, permission, and unbounded-loop risks while
  reporting only paths, counts, codes, severities, and line hashes.
- The generic proactive workflow has a weekly scheduled path plus a safe
  `.gitclaw/proactive/repo-hygiene.md` default prompt.
- Proactive enqueue supports `--not-before` due gates for reminders: before
  due it outputs `issue_number=0`, `due=false`, `skipped=true` with no writes.

## Runtime Notes

- Read-only tools include bounded list/search/read, skill-index, and policy
  metadata.
- Search-result verification tokens must come from `gitclaw.search_files`, not
  from issue-thread nonce echoes.
- Search has per-query match caps so broad file/path queries cannot hide later
  explicit phrase matches.
- `@gitclaw /tools info <name>` and `gitclaw tools info <name>` expose one
  tool contract plus active-output hashes without raw tool inputs or outputs.
- `/policy verify` and local `policy verify` check main workflow job
  permissions against the read-only contract without raw policy bodies.
- Channel ingest dedupes repeated `channel + message_id`; dispatch wakeups use
  matching channel-message text; offsets, leases, and receipts are hashed.
- Skills are loaded with progressive disclosure; full `SKILL.md` bodies are
  prompt inputs only when selected or marked always-on.
- Tool outputs are repo-gated like skills: `tools.allowed` and
  `tools.disabled` constrain deterministic built-ins while reports still show
  enabled/disabled/allowlist-blocked state.
- `.gitclaw/config.yml` is read-only, schema-checked repo config. Defaults load
  first, repo config second, and environment overrides last.
- GitHub Models can return bursty `429` responses during dense live E2E runs;
  the client retries transient `429`, timeout, and `5xx` failures with bounded
  delay and request timeout.
- The primary model remains `openai/gpt-5-nano`, but repo config can declare
  fallback models such as `openai/gpt-4.1-nano`; assistant markers record the
  actual model that answered, and invalid-model E2Es disable fallback.
- Normal LLM assistant markers should carry body-free prompt provenance:
  prompt context hash, context/skill/tool counts, selected skill names, and
  prompt-visible tool names so live E2E can prove actual tool usage.
- `/session` should surface assistant-turn prompt provenance from prior model
  replies: counts, unique prompt-context hashes, and per-turn skill/tool names
  without raw prompts, tool outputs, or assistant bodies.
- `/skills risk` should scan repo-local `SKILL.md` bodies internally for risky
  prompt-boundary, persistence, host-exec, exfiltration, and unbounded-loop
  instruction categories while reporting only counts, codes, paths, and line
  hashes.
- Workflows use Node 24-compatible first-party actions: `actions/checkout@v5`,
  `actions/setup-go@v6`, and `actions/upload-artifact@v6`.
- Artifact governance is body-safe: `.gitclaw/ARTIFACTS.md` and
  `.gitclaw/artifacts/*.md` are metadata/report inputs, and `/artifacts`,
  `/artifact`, `gitclaw artifacts list`, and `gitclaw artifacts verify` audit
  upload-action version, retention, redaction, approval, and label gates without
  printing artifact bodies.
- `@gitclaw /artifacts risk` and local `gitclaw artifacts risk` scan artifact
  policy, prompt-artifact specs, and upload workflows for prompt-boundary,
  credential, unredacted prompt, raw payload logging, hidden-state, external
  storage, retention, label-gate, missing-file, approval/redaction, repository
  mutation, and unbounded-collection risks while reporting only metadata,
  codes, severities, and line hashes.
- Diff governance is body-safe: `.gitclaw/DIFFS.md` and
  `.gitclaw/diffs/*.md` are metadata/report inputs, and `/diffs`, `/diff`,
  `/changes`, `gitclaw diffs summary`, `gitclaw diffs risk`, and
  `gitclaw diffs verify` audit git status, changed paths, numstat totals, and
  diff risk boundaries without printing patch hunks or file bodies.
- `@gitclaw /diffs risk` and local `gitclaw diffs risk` scan diff policy,
  working-tree specs, and git metadata for prompt-boundary, credential, raw
  patch leakage, destructive git action, hidden-state, untracked-file body,
  external storage, approval, unsafe raw-patch mode, and unbounded collection
  risks while reporting only metadata, paths, codes, severities, and line
  hashes.
- Workspace governance is body-safe: `.gitclaw/WORKSPACE.md` and
  `.gitclaw/workspaces/*.md` are metadata/report inputs, and `/workspace`,
  `/workdir`, `/repo`, `gitclaw workspace summary`,
  `gitclaw workspace risk`, and
  `gitclaw workspace verify` audit the GitHub Actions checkout, repository
  inventory counts, context allowlist, workflow checkout/setup-go versions, and
  no-private-memory boundaries without printing file or workflow bodies.
- `@gitclaw /workspace risk` and local `gitclaw workspace risk` scan
  workspace policy, workspace specs, and workflow metadata for prompt-boundary,
  credential, private-memory, external-mount, destructive-mutation,
  long-running-service, raw-body, checkout/setup/fetch-depth, approval, and
  unbounded-inventory risks while reporting only metadata, paths, counts, codes,
  severities, and line hashes.
